Leros Server / Worker 与 Agent 执行架构设计说明

本文档描述 Server（控制面）、Worker（执行面）以及独立 Agent 执行层的当前职责边界。2026-06-28 的架构迁移已经落地，主链、目录和契约均以当前代码为准。

文档状态：当前实施架构。实施结果、测试证据和维护项见 Agent 执行架构实施进度；原始决策依据见 架构重整方案。

Leros 采用 Server（控制面）- Worker（执行面） 分离架构，通过 NATS JetStream 解耦通信。Worker 内部进一步分为通信调度、SingerOS 业务包装和独立 Agent 执行三层。

二、架构层级详解

2.1 进程入口层（cmd）

cmd 进程入口

对应文件: backend/cmd/leros/server.go, backend/cmd/leros/worker.go

进程入口层通过 Cobra CLI 启动两个独立的进程：

  • Server 进程（leros server）—— 控制面，监听 :8080，承载 HTTP API、事件发布、数据库连接。负责接收前端请求，将任务通过 NATS 下发给 Worker。
  • Worker 进程（leros worker [codex|claude|opencode]）—— 执行面，监听 :8081，订阅 NATS 命令执行 Agent Run。通过 NATS JetStream 消费 Server 下发的任务。

该层的职责边界（依据项目约束）：

  • ✅ Cobra 命令注册、进程生命周期管理（lifecycle.Std().WaitExit()）、信号处理、log.Fatal
  • ❌ 任何业务逻辑

Server 与 Worker 是两个独立部署的二进制，通过 NATS 集群通信，天然支持水平扩展 Worker。

2.2 API / 路由层

API HTTP API & 路由

对应文件: backend/internal/api/handler/, backend/internal/api/middleware/, backend/internal/worker/router/router.go

Server 端 API 注册 — api.SetupRouter() 在 Gin 上注册所有 REST 端点：

  • Session 管理（创建 / 列表 / 消息）
  • Project 管理（CRUD + 文件树）
  • Digital Assistant 管理
  • Skill Marketplace（查询 / 安装 / 卸载 / 下载）
  • Connector 管理（GitHub 等外部集成）

Worker 端路由注册 — router.SetupRouter() 注册 Worker 本地 HTTP 端点：

端点作用
GET /health健康检查
/v1/mcpMCP 协议端点（Agent 通过 MCP 调用 Worker 的工具能力）
/v1/chat/completionsOpenAI Chat Completions 模型路由（modelrouter v2）
/v1/messagesAnthropic Messages 模型路由
/v1/responsesOpenAI Responses 模型路由

2.3 Service 层（业务逻辑）

Service 业务逻辑层

对应文件: backend/internal/service/

Server 端的核心业务编排层，处理会话管理、项目操作、Skill 安装/卸载等业务逻辑。该层通过 infra/mq 的 EventBus 向 NATS JetStream 发布 WorkerCommand，将 Agent 执行任务下发给 Worker。

关键类型 WorkerCommand（定义于 backend/pkg/messaging/command.go）：

type WorkerCommand struct { ID string // 消息唯一 ID Type MessageType // "worker.command" CreatedAt time.Time Trace TraceContext // {RequestID, TaskID, RunID} Route RouteContext // {OrgID, WorkerID, SessionID} Body WorkerCommandBody // 根据 CommandType 携带不同 Payload Metadata json.RawMessage // 强类型命令元数据的 JSON } type WorkerCommandBody struct { CommandType CommandType // agent.run | run.cancel | approval.resolve | question.answer | skill.manage Payload json.RawMessage // 对应类型的 JSON payload ReplyTo string // NATS reply inbox（用于同步回复，如 skill 管理） }

五种命令类型分别路由到不同的 NATS subject（Lane）：

CommandTypeLaneSubject 模式说明
agent.runcmd.runorg.{org_id}.worker.{worker_id}.cmd.runAgent 执行任务
run.cancelcmd.controlorg.{org_id}.worker.{worker_id}.cmd.control取消正在运行的任务
approval.resolvecmd.interactionorg.{org_id}.worker.{worker_id}.cmd.interaction审批决策回传
question.answercmd.interaction（同上）问题答案回传
skill.managecmd.skillorg.{org_id}.worker.{worker_id}.cmd.skillSkill 安装/列表/卸载

2.4 消息中间件层（infra/mq）

infra/mq NATS JetStream 事件总线

对应文件: backend/internal/infra/mq/

基于 NATS JetStream 实现 EventBus 接口，提供持久化消息队列：

  • Publish — Server 向 NATS subject 发布 WorkerCommand
  • Subscribe — Worker 通过持久化 Consumer 订阅对应 subject，保证消息不丢失
  • Request/Reply — 同步请求-响应模式，用于 Skill 管理等需要同步返回的场景（通过 core NATS inbox 机制）

Worker 端的 NATS 连接同时用作 JetStream Consumer（任务消费）和 core NATS（reply 回复）。Server 端用作 Publisher 和 reply Subscriber。

2.5 Worker 命令分发层（command）

command 统一命令分发器

对应文件: backend/internal/worker/command/dispatcher.go

Dispatcher 是 Worker 端统一的 NATS 命令入口，负责四条 lane 的订阅、WorkerCommand 解析和 handler 分发。任一 lane 的订阅异常退出时，Dispatcher 会取消其他 lane 并向进程入口返回错误：

  • Lane 机制：启动后四个独立 goroutine 分别订阅四个 NATS subject lane，反序列化为统一的 WorkerCommand，根据 command_type 分发到对应 handler。
LaneHandler 接口实现者处理逻辑
cmd.run RunHandler command/run.Handler 解码 RunCommandPayload → 校验 → 防抖合并 → WorkerPool 执行 → Seq 追踪
cmd.control ControlHandler command/run.Handler 取消正在运行的 Agent Run（通过 context.CancelFunc）
cmd.interaction InteractionHandler command/interaction.Handler 审批决策 / 问题答案 → 调用注入 Runtime 的 agent.InteractionHandler 唤起阻塞执行
cmd.skill SkillHandler command/skill.Handler Skill install/list/detail/uninstall/import → 通过 core NATS reply 返回结果

Dispatcher 接收原始 *nats.Msg，使 handler 能获取 stream seq 用于 crash recovery（seq tracking）和 背压控制（NATS ACK）。

2.6 Worker Run 适配与调度层

worker Command Adapter + RunCoordinator

对应目录: backend/internal/worker/command/run/, backend/internal/worker/run/

Worker Run 层只处理通信与进程内调度，不生成 prompt，也不理解 Model、Skill、Memory、Workspace 或 Runtime 协议。

Command Adapter 职责：

  • 解码并校验 WorkerCommand、route 与 payload。
  • 读取 JetStream delivery sequence 并映射为 RunSubmission。
  • 调用 Coordinator；在批次执行完成后更新 delivery 状态。

RunCoordinator 职责：

  • Session 级 1500ms debounce 与消息合并。
  • 并发限制、pending waiter、active run 与 graceful close。
  • 通过 context.CancelFunc 处理 cmd.control。
  • 聚合 delivery sequence，但不依赖 NATS subject 或 payload 类型。
当前状态：Coordinator 已接入主链。每个 debounce waiter 等待真实批次结果；同 Session 串行、不同 Session 并行；Handler 只在批次完成后更新 delivery sequence。

2.7 SingerOS Agent 业务包装层

assistant 业务 Run 编排

对应目录: backend/internal/assistant/

assistant.Service 是一次 SingerOS 业务 Agent Run 的唯一编排入口，包裹独立执行层：

  • Preparer：读取 Session history、Memory 和 Skill，解析模型、Tool、Workspace 与附件，构造纯执行层 ExecutionRequest。
  • Service：发布 run.started，调用 agent.Executor，分类 completed/failed/cancelled，并保证每次 Run 只有一个终端事件。
  • Finalizer：执行 Workspace reconciliation、Artifact 收集和 Git stage/commit/push。
  • Journal：归档完整 activity event、usage、tool call 和统计信息，供终端事件持久化。

Assistant、Actor、Org、Project、Skill、Memory、Artifact 和 Git 都只能存在于本层及其业务依赖中，不得进入独立 Agent Runtime。

2.8 独立 Agent 执行层

backend/agent Executor + Runtime

对应目录: backend/agent/, backend/agent/runtime/

backend/agent 是顶层可复用执行包，不依赖 SingerOS 的 internal/*、config、messaging、NATS、Session、Memory、Skill 或 Workspace。

type Runtime interface { Name() string Execute( ctx context.Context, request ExecutionRequest, observer Observer, ) (ExecutionResult, error) }

Executor：

  • 校验已经准备完成的 ExecutionRequest。
  • 通过 Registry 按 name 解析 Runtime。
  • 统一执行生命周期、取消与 Observer 错误处理。
  • 返回不包含业务 RunStatus 的 ExecutionResult。

四个同级 Runtime：

  • native：内部使用 Eino，但只消费 ExecutionRequest。
  • claude、codex、opencode：负责各自 CLI 或服务协议。
  • 所有 Runtime 使用同一个接口、事件模型、取消语义和 contract test。

执行边界：

  • Runtime 只能产生 message、reasoning、tool、approval、question 和 provider-session 活动。
  • Runtime 不访问 NATS、数据库，不生成业务 run.* 终态，不执行 Git。
  • Tool 使用具名类型或 json.RawMessage，公共签名禁止 map[string]interface{}。
  • Approval/Question 通过构造时注入的强类型 InteractionHandler 处理。

2.9 Run Event 投影层

projector Server 端状态与流投影

对应文件: backend/internal/runnable/session_run_state_projector.go, session_run_stream_projector.go

Worker 通过 assistant.EventMapper 将 core execution event 转为 messaging.RunEvent，再由 event publisher 按事件语义发布到两条 lane：

  • run.stream：message、reasoning、tool、todo 等高频增量事件，供 SSE 实时推送和回放。
  • run.state：run started/terminal、artifact、approval、question 等低频关键状态。

Server 分别启动 stream projector 与 state projector。两条 lane 的 JetStream sequence 独立记录：StreamStartSeq 只能来自首条 run.stream，StateStartSeq 来自 run.started。

2.10 Worker 实例调度层（scheduler/）

scheduler Worker 实例调度

对应文件: backend/internal/worker/scheduler.go, backend/internal/worker/scheduler/

定义 WorkerScheduler 接口，管理 Worker 进程/容器的生命周期：

type WorkerScheduler interface { Start(ctx context.Context, spec *WorkerSpec) (*WorkerInstance, error) Stop(ctx context.Context, workerID string) error Health(ctx context.Context, workerID string) error List(ctx context.Context) ([]*WorkerInstance, error) }

三种实现（通过 factory 按配置选择）：

SchedulerEnvType说明
ProcessSchedulerprocess本地进程模式：fork leros worker 子进程，管理 PID 生命周期
DockerCLISchedulerdocker-cliDocker 容器模式：通过 docker run 启动 Worker 容器
KubernetesSchedulerkubevirtK8s 模式：通过 Kubernetes API 创建 Pod 运行 Worker

Worker Server（server/server.go）通过 WebSocket 管理 Worker 的连接、心跳、注销，并暴露 CreateWorker API 供外部动态创建 Worker 实例。

3.2 逐步详解

Step 1 — 前端请求到达

用户发送消息 → POST /v1/sessions/:id/messages → api/handler/ 中的 Session Handler 处理。

Step 2 — Server 构建 WorkerCommand

Service 层调用 messaging.NewRunCommand() 构建 WorkerCommand，包含完整上下文：Assistant 配置、Model 参数、Actor 身份、Workspace 信息、Input 消息列表。

Step 3 — NATS 发布

Server 通过 EventBus.Publish() 将 JSON 序列化的命令发布到 org.{org_id}.worker.{worker_id}.cmd.run subject，JetStream 持久化存储。

Step 4 — Worker 消费

Worker 的 command.Dispatcher 为每条 lane 启动独立 goroutine，通过 JetStream Consumer 拉取消息，解析为 WorkerCommand，并路由到 command/run.Handler。

Step 5 — Worker 适配与调度

Command Adapter 完成 wire 校验和 RunSubmission 映射；RunCoordinator 负责 seq 聚合、Session debounce、并发、等待和取消，然后调用 assistant.Service。

Step 6 — 业务准备与独立执行

assistant.Preparer 读取 Session、Memory、Skill、Model 和 Workspace，生成纯 agent.ExecutionRequest。agent.Executor 选择 native、Claude、Codex 或 OpenCode Runtime。Runtime 只返回 execution event 和 result。

Step 7 — 业务收尾与结果回传

assistant.Finalizer 处理 Artifact 和 Git，Service 决定唯一业务终态。worker/eventpub.NATSEventSink 将 core event 转为兼容的 messaging.RunEvent，分别发布到 run.stream 或 run.state。Server projector 持久化关键状态并提供 SSE 实时流与回放。

四、关键设计决策

设计点决策原因
消息协议 统一 WorkerCommand envelope 所有 Server→Worker 通信使用同一种消息格式，通过 CommandType 区分类型，避免多套协议维护成本
Lane 分发 四条独立 subject，分别 goroutine 消费 不同命令类型隔离消费，互不影响；高优先级的 control（cancel）不会被 run 的队列阻塞
防抖合并 Session 级别 1500ms 窗口 用户连续打字会产生多条消息，防抖合并后只触发一次 Agent Run，减少 LLM API 调用成本
Seq Tracking SQLite 持久化 NATS stream seq Worker crash 重启后不会重复执行已处理的任务；提升系统可靠性
结果回传 Core Event → EventMapper → 双 lane NATS Core 与 wire event 明确分离；stream/state 分流继续兼顾实时推送、状态持久化和独立 replay sequence
多 Runtime 统一 agent.Runtime + Registry native / Claude / Codex / OpenCode 完全同级，使用同一请求、事件、结果和取消契约
审批/提问 注入强类型 InteractionHandler 保留交互能力，但 Runtime 不依赖 NATS 或 package-level 全局 router
执行与业务 backend/agent + internal/assistant 执行层不包含 SingerOS 业务；业务层负责 Session、Memory、Skill、Workspace、Artifact 和 Git
Runtime 输入 准备完成的 ExecutionRequest Runtime 不再接收同时承担原始输入和跨阶段可变状态的 RequestContext
Worker 调度 Process / Docker / K8s 三模式 适应不同部署环境：开发用 Process，生产用 Docker 或 K8s

六、实施状态与维护约束

原计划中的 P0、P1、P2 架构项已经实施。完整代码证据和验证结果见 AGENT_EXECUTION_ARCHITECTURE_PROGRESS_2026-06-28.md。

完成 主链与调度语义

唯一主链为 command/run.Handler → RunCoordinator → assistant.Service → agent.Executor → Runtime。Coordinator 已实现 debounce 全 waiter、并发上限、Session 串行、取消、Close 和 delivery sequence 完成/失败语义。

完成 独立执行边界

backend/agent 递归禁止依赖 internal/*、config、messaging 和业务 Tool 包。PreparedRun、Workspace、Session、Memory、Skill、Artifact、Git 与业务 RunStatus 均由 internal/assistant 持有。

完成 Runtime 与事件收敛

native、Claude、Codex、OpenCode 作为具体 Runtime 注册；native 直接执行 Eino，三个 CLI Runtime 复用 externalcli Driver。主路径只有一套 ExecutionRequest、ExecutionResult 和 agent.Event；provider、events、eventpub 和 todo 的迁移期 type alias 已删除。

完成 旧层和全局状态清理

backend/engines、internal/runtime、internal/agent、eventengine 和 simplechat 已退出代码树。Memory、Model、Skill mutation、Interaction、provider session 与 MCP token 均由 composition root 构造并注入。

维护约束 防止边界回退

新增 Runtime 必须实现同一 contract suite；新增 Tool 的公共输入必须为 json.RawMessage 或具名结构；Runtime 不得发布业务 run.* 终态；外部事件必须转换为 WorkerCommand 后进入唯一主链。
